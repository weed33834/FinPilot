## Design Philosophy

FinPilot AI abstracts the real workflows of enterprise finance teams into a four-stage closed loop: **Perceive → Reason → Act → Audit**.

- Multi-role agents orchestrated by **LangGraph** handle analysis, modeling, debate, and synthesis.
- Financial computation runs through **deterministic code paths** (DCF, WACC, comparables, backtesting); the LLM is restricted to narrative and explanation.
- Every API call, every question and answer, and every module toggle is written to a **runtime log**, so the whole process is traceable and auditable.

> Numbers are computed by code, narratives are generated by the model, and every line of output can be traced to its source.

## Core Capabilities

| Module | Key capabilities |
| :--- | :--- |
| 🤖 **Q&A agent** | Conversational upload of Excel / PDF / CSV / DOCX, automatic parsing and context injection, **SSE streaming** that pushes ReAct reasoning steps in real time |

| 🛡 **Security & compliance** | ABAC access control, TOTP two-factor auth, PII masking, injection protection, audit logs, and **role-tiered permissions** |
| 📡 **Runtime logs** | Four tabs (log list / Q&A interactions / module status / statistics dashboard) for real-time monitoring |
| 🧰 **Extension system** | Tools, skills, MCP servers, code sandbox, and prompt management under a unified integration |
| 💬 **Conversation as control** | A **slash-command system** that exposes all features from the chat surface, filtered by role |
| 🚨 **Granular errors** | **FetchError + level-based highlighting** — a four-color alert for network / auth / client / server errors |

## Architecture

<div align="center">
  <img src="docs/architecture.svg" alt="FinPilot AI system architecture" width="860" />
</div>

The system is organized into three layers:

- **Presentation layer** — React 19 + Vite SPA with chat / report / audit / admin panels and real-time SSE push.
  - `AgentChatPage` integrates the slash-command palette, a level-based error bar, and streamed reasoning steps.
  - `MarkdownRenderer` ships with XSS sanitization and code-block syntax highlighting.
  - `SlashCommandPalette` offers fuzzy search, keyboard navigation, and role filtering.
- **Service layer** — FastAPI + LangGraph for routing, auth, agent orchestration, parsing, retrieval, computation, and audit logging.
  - The ReAct agent uses `agent.stream(stream_mode="updates")` to push each node's state in real time.
  - Multiple LLM output formats are tolerated: the standard ReAct three-part form, `<tool_call>` XML, and `<answer>` tags.
  - LLM configuration is read from the database first (maintained in the admin console), falling back to environment variables.
- **Data layer** — SQLite (ORM), vector store, BM25 inverted index, and file/config storage.
  - ReAct checkpoints support two backends: `memory` (default) and `sqlite` (persistent).

## ReAct Workflow

<div align="center">
  <img src="docs/workflow.svg" alt="ReAct workflow" width="860" />
</div>

The ReAct loop runs up to 5 rounds of tool calls, with each completed node pushed in real time over SSE:

- **start** — session created
- **thinking_token** — agent reasoning / tool call / observation result
- **answer_token** — incremental final answer
- **done** — terminal event carrying the reasoning chain, confidence, and intent
- **error** — exception event with detailed error information

**Heartbeat guard**: if no event is pushed within 15 seconds, `…` is emitted to prevent the frontend from misjudging a timeout.

## Conversation as Control Hub

The chat surface is central to FinPilot. Administrators can invoke any feature through slash commands from the conversation box, controlling the whole program, application, and agents. Regular users can only invoke commands within their permitted scope.

### Slash-command palette

Typing `/` in the input box opens the command palette, supporting fuzzy search, keyboard navigation, and grouping by category. All commands are filtered by role:

| Category | Example commands | Role |
| :--- | :--- | :--- |
| **help** | `/help`, `/?` | All users |
| **data** | `/dashboard`, `/queries history`, `/conversations list`, `/documents list` | user |
| **report** | `/reports list`, `/reports generate 600519 Kweichow Moutai`, `/reports status <task_id>` | user |
| **analysis** | `/factor categories`, `/backtest strategies` | user |
| **system** | `/admin status`, `/admin health`, `/models list`, `/models test <provider_id>` | admin |
| **admin** | `/users list`, `/audit logs`, `/approvals list`, `/templates list`, `/subscriptions list` | admin |

### Permission tiers

- **Administrator (admin)**: can invoke all 19 commands, covering the five categories of data / research / analysis / system / administration.
- **Regular user (user)**: can invoke only the four categories of help + data + report + analysis, totaling 9 commands, and cannot access sensitive operations such as system status, user management, or audit logs.
- The backend `require_admin` dependency re-validates every admin command endpoint; frontend filtering is a UX optimization only.

## Error System

FinPilot's error system is designed for **precise localization and clear visibility**, replacing generic fallbacks such as "Operation failed, please try again later."

### Error levels and colors

Each error displays a colored alert light with a pulse animation, glow, and gradient background so it stays legible under both light and dark themes:

| Level | Color | Trigger |
| :--- | :--- | :--- |
| `network` | Gray | Connection timeout, DNS failure, CORS rejection, backend not started |
| `auth` | Yellow | 401 not logged in, 403 insufficient permission |
| `client` | Orange | 400/404/422 request parameter errors, route not found |
| `server` | Red | 500/502/503 server internal error |
| `unknown` | Red | Unclassified fallback |

### Error message format

The error bar shows the specific failing endpoint, HTTP method, status code, and backend-returned detail, for example:

```
[POST /agent/chat/stream] 500 server internal error — KeyError: 'react_steps'
[network] request timeout (30s) — backend did not respond in time; possibly slow LLM inference or backend blocking
[GET /model-configs] 422 validation failed — body.question: field required
```

### Implementation notes

- `FetchError` class: carries `status` / `url` / `method` / `bodyText` / `code`, so SSE endpoints called via `fetch` (not axios) can reuse the unified error system.
- `getErrorLevel(err)`: infers the error level from FetchError / AxiosError / DOMException / TypeError.
- `getErrorMessage(err)`: converts the raw error into a precise string with a source tag, status code, and backend reason.

## Quick Start

### 1. Clone the repository

```bash
git clone https://github.com/weed33834/FinPilot.git
cd FinPilot
```

> GitCode mirror: `git clone https://gitcode.com/badhope/FinPilot.git`

### 2. Prepare the Python environment

```bash
python3 -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate
pip install -e .
```

> Requires Python 3.10–3.13. Managing multiple versions with [pyenv](https://github.com/pyenv/pyenv) is recommended.

### 3. Configure environment variables

Copy the example file and adjust as needed (all variables are optional):

```bash
cp .env.example .env
```

The minimal configuration only requires an LLM provider. FinPilot reads LLM configuration **from the database first** (maintained under Admin → LLM Providers), with environment variables as a fallback.

**Option A: environment variables (quick trial)**

```bash
export OPENAI_API_KEY="sk-..."
# Optional:
export OPENAI_BASE_URL="https://api.openai.com/v1"
export OPENAI_MODEL="gpt-4o-mini"
```

**Option B: MoonWeaver (OpenAI-compatible protocol)**

```bash
# In Admin → LLM Providers, create:
#   name=MoonWeaver, provider_type=openai, base_url=https://api.587.lol/v1
#   api_key=any, is_default=true
#   models: moonweaver-4.8 (only this model is currently offered by the API; can be attached to both high/low tiers)
```

Anthropic is also supported:

```bash
export ANTHROPIC_API_KEY="sk-ant-..."
export ANTHROPIC_MODEL="claude-3-5-sonnet-20241022"
```

### 4. Start the backend

```bash
uvicorn finpilot_equity.web_app.main:app --host 0.0.0.0 --port 8001
```

On first launch, the database is created automatically and a default administrator account is initialized:

| Field | Value |
| :--- | :--- |
| Username | `admin@finpilot.ai` |
| Password | `admin123` |

> ⚠️ Change the default password in User Management immediately after first login.

### 5. Start the frontend

```bash
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173` in a browser and log in with the default administrator account.

### 6. Containerized deployment (optional)

```bash
docker build -t finpilot-ai:1.0.0 .
docker run -d \
  -p 8001:8001 \
  --env-file .env \
  --name finpilot finpilot-ai:1.0.0
```

See [docs/DEPLOYMENT.md](docs/DEPLOYMENT.md) for detailed deployment guidance.

## Runtime Logs Module

The settings section includes a Runtime Logs module that monitors the running state of the entire pipeline in real time:

| Tab | Purpose |
| :--- | :--- |
| **Statistics dashboard** | Total log count, new today, success rate, aggregated module enablement status |
| **Log list** | Category / level / source / duration / status code / payload detail for each API call |
| **Q&A interactions** | Aggregated by session, replaying user questions and agent answers |
| **Module status** | Enablement statistics for modules such as LLM / tools / skills / sandbox / MCP / RAG / Text2SQL |

Logging is best-effort: a write failure does not affect the main flow, and logs can be exported to CSV for offline analysis.
